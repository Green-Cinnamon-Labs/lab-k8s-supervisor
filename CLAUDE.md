# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Purpose

`tep-supervisor` is the infrastructure and orchestration layer for the Tennessee Eastman Process (TEP) Digital Twin Lab. It manages the local Kind cluster, Kubernetes manifests, and cloud environment scripts that tie together the other three repos: `tep-plant` (Rust simulator), `tep-operator` (Go/Kubebuilder controller), and `tep-ihm` (Python dashboard).

## Architecture

The local lab runs two separate execution contexts:

- **Docker Compose** (`te-plant`, `tep-ihm`) — represent external systems, not managed by Kubernetes
- **Kind cluster** (`plc-operator` Pod) — Kubernetes controller that reconciles `PLCMachine` CRDs

The operator connects to the plant via `host.docker.internal:50051` (gRPC). This host alias is resolved by Docker Desktop and allows the in-cluster operator to reach the plant container running outside Kind.

## Key Files

- `local/docker-compose.yml` — defines `te-plant` (port 50051) and `tep-ihm` (port 8080); environment variables control step delay and disturbance injection (`ACTIVE_IDV`)
- `local/setup.sh` — idempotent script: creates Kind cluster, loads operator image, applies `k8s/crd.yaml` + `k8s/operator-deployment.yaml` + `k8s/plcmachine-sample.yaml`
- `local/k8s/crd.yaml` — `PLCMachine` CRD (API group `infrastructure.greenlabs.io`), generated by controller-gen from `tep-operator`
- `local/k8s/operator-deployment.yaml` — operator Deployment + ServiceAccount + ClusterRole; uses `imagePullPolicy: Never` (requires prior `kind load docker-image`)
- `local/kind-config.yaml` — single control-plane Kind cluster named `tep-lab` on API port 6443

## CRD Updates

When `tep-operator` regenerates `crd.yaml`, copy it into this repo:

```bash
cp ../tep-operator/config/crd/bases/infrastructure.greenlabs.io_plcmachines.yaml local/k8s/crd.yaml
```

## Simulation Parameters

`STEP_DELAY_MS=36` in docker-compose accelerates the TEP simulation 100×: 1 simulated hour = 36 real seconds. `ACTIVE_IDV` sets the integer disturbance ID (leave unset for nominal operation).

## Agent Rules

- Do not run Docker, Kind, kubectl, Terraform, cloud, teardown, or environment-changing commands unless explicitly authorized.
- Prefer suggesting operational commands and explaining their purpose instead of executing them.
- Treat `local/` as the active development environment.
- Treat `k8s-lab-1-aws/` as legacy; do not modify it unless explicitly requested.
- Treat `k8s-lab-1-azr/` and `k8s-lab-1-gcp/` as placeholders; do not expand them without explicit direction.
- Do not manually edit `local/k8s/crd.yaml` unless the user asks for a temporary patch.
- The canonical CRD source is `tep-operator`; when CRDs change, copy regenerated manifests from `tep-operator`.
- Do not change image names, ports, namespaces, or `host.docker.internal:50051` assumptions without reporting cross-repo impact.
- Preserve the separation: plant and IHM run in Docker Compose; operator runs inside Kind.
- Before editing scripts or manifests, provide a short plan and list the files to be modified.
- For destructive operations such as `docker compose down`, `kind delete cluster`, or Terraform destroy, ask for explicit confirmation.
- For cross-repository decisions, follow the root `CLAUDE.md` / `AGENTS.md` propagation rules.
